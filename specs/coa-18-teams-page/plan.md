# Implementation Plan: COA-18 Teams Page

**Branch**: `coa-18-teams-page` | **Date**: 2026-04-12 | **Spec**: `specs/coa-18-teams-page/spec.md`

---

## Summary

This plan builds the Teams section of the Bendigo Phoenix website in three phases. P1 delivers a fully functional listing page backed by a static JSON file of placeholder team data. P2 extends the scraper to fetch real PlayHQ team data, writes per-team fixture and ladder files, and builds the team detail pages. P3 adds a game detail modal to the fixture table on detail pages.

The implementation follows the existing data-file pattern established by `scores.astro` and `scrape-playhq.js`: a Node script fetches data from the PlayHQ REST API, writes JSON files to `scripts/`, and the Astro pages read those files at build time with a hardcoded fallback.

---

## Technical Context

- **Language/Runtime**: Node.js (existing scraper scripts), Astro (static site)
- **Primary Dependencies**: Astro (pages and components), Tailwind CSS (styling), vanilla JavaScript (client-side filter and modal)
- **Storage**: Static JSON files in `scripts/` (generated, PlayHQ-sourced) and `src/data/teams/` (committed, manually maintained)
- **Testing**: Manual acceptance-criteria walkthroughs per phase (no automated test framework in this project)
- **Target Platform**: Web (static Astro site, served as HTML)
- **Performance Goals**: Listing page renders within 2s; filter interactions respond within 100ms (client-side only, no network)
- **Scale/Scope**: Up to ~21 Phoenix teams across up to 2 active seasons; must not assume a fixed count

- **Principle I (User Outcomes)**: PASS. Each phase has observable, independently testable outcomes tied to real user needs.
- **Principle II (Test-First)**: PASS with project context. This is a static Astro site with no automated test runner. Acceptance criteria from the spec serve as the test specification; each phase is independently verifiable before the next begins.
- **Principle III (Backend Authority)**: PASS. Data authority lives in PlayHQ-generated JSON files; the client reads and renders only. The filter JS hides/shows DOM nodes — no data inference or recalculation occurs client-side.

- **Principle IX (Cross-Feature Consistency)**: PASS. Filter bar pattern copied from `team.astro` (`sticky top-16 z-40`, pill buttons). Data-file pattern copied from `scores.astro`. ## Open Questions — Resolved for Implementation

The spec listed six open questions. Resolutions for each before work begins:

1. **Filter state persistence**: Use `sessionStorage`. A URL query param (`/teams?filter=boys`) is more robust and shareable, but adds complexity and changes the URL seen by the back button. `sessionStorage` is simpler, already used for this pattern in the existing site, and restores the filter correctly when the user presses back. Store a single key `teams-filter` as a JSON object `{ type, value }`.

2. **Design tokens for pills**: Use the existing token set. Division pill: `bg-brand-purple text-white`. Game Night pill: `bg-brand-gold text-white`. Boys/Girls pill: `bg-gray-700 text-white` (Boys) and `bg-purple-200 text-brand-purple` (Girls). TBC state: `bg-gray-200 text-gray-500` for all pill types. Contrast to be verified against WCAG AA before shipping P1.

3. **Page hero**: Yes, include a hero section matching the `team.astro` / `scores.astro` pattern. Sub-headline: "Bendigo Phoenix Juniors" with a gold divider. This is consistent with every other page in the site.

4. **Teams data source for P1**: Store placeholder data in `src/data/teams/teams.json` (committed to the repo). This is consistent with other data files in `src/data/`. The Astro page imports it directly (no `readFileSync` needed for this file — it is committed, not generated).

5. **Game detail modal scope (P3)**: Use data already in the per-team fixture file — no runtime API call. All game data (opponent, venue, date, time, scores, home/away status) is written to `scripts/teams-data.json` at scrape time. The modal reads from the DOM or from a data attribute on the game row.

6. **Pagination vs scroll**: No pagination in scope. The listing grid scales by count naturally (CSS grid, no hardcoded counts). If teams grow significantly, an infinite scroll or pagination can be added later without breaking the existing structure.

## Grade Name Parsing Logic

All four derived fields are extracted from the PlayHQ grade name string. This logic lives in `scrape-teams.js` and is also replicated inline in the Astro page for P1 display from static data.

```
Input:  "Monday U14 Girls 4"

Rules:
  gameNight = first token if it is a day-of-week name
              ["Monday","Tuesday","Wednesday","Thursday","Friday","Saturday","Sunday"]
              → "Monday"

  ageGroup  = token matching /^U\d+$/i (case-insensitive), lowercased + "s"
              → "u14s"
              Unknown age group → place team in "Other" section

  gender    = token "Boys" or "Girls" (exact, case-insensitive)
              → "Girls"
              Absent → null (pill shows "TBC")

  division  = last numeric token → "DIV 4"
              Absent → null (pill shows "TBC")
```

Edge cases:
- No day token → `gameNight = null` (pill shows "TBC")
- No gender token → `gender = null` (pill shows "TBC")
- No division number → `division = null` (pill shows "TBC")
- Unknown age group → `ageGroup = "other"` (placed in "Other" section, not dropped)

## Slug Generation

```
Input team name:  "PHOENIX Roughnuts"
Grade name:       "Monday U14 Girls 4"

Slug = [name-slug]-[ageGroup]-[gender]-[division]
     = "phoenix-roughnuts-u14s-girls-div-4"

Steps:
  1. Lowercase the team name
  2. Replace all non-alphanumeric characters with "-"
  3. Collapse multiple "-" into one
  4. Append ageGroup, gender (lowercased), division (lowercased, space→"-")
  5. Trim leading/trailing "-"

Collision handling: if two teams produce the same slug,
  append the first 8 characters of the PlayHQ team ID:
  "phoenix-roughnuts-u14s-girls-div-4-b3efb4fc"
```

## `scrape-teams.js` — How It Works (P2)

The new script shares the `apiFetch` and `fetchAllPages` helpers from the existing scraper pattern (or imports them from a shared module if the team decides to refactor). It is an independent script, not a modification of `scrape-playhq.js`.

```
1. For each season ID in SEASON_IDS:
   a. GET /v1/seasons/{seasonId}/teams?pageSize=100
      Paginate using fetchAllPages.
   b. Filter: team.name.toLowerCase().includes('phoenix')
   c. For each Phoenix team:
      - Parse gradeName using the grade name parsing rules above
      - Generate slug (with collision check across all teams)
      - GET /v1/grades/{team.gradeId}/games  → fixture array
        Filter: all games (not just Phoenix; the team IS Phoenix)
        Normalise each game into the fixture shape
      - GET /v1/grades/{team.gradeId}/ladder → ladder array
        Find the Phoenix team row by name match
        Extract: position, wins, losses, draws, pointsFor, pointsAgainst, points

2. Write scripts/teams-data.json
```

The script uses the same API key, tenant, and org constants as `scrape-playhq.js`. It reads `PLAYHQ_API_KEY` from `process.env` or falls back to a placeholder with the same guard and error message pattern.

Run command: `node scripts/scrape-teams.js`

## Phased Delivery

### Phase 1 — Teams Listing Page with Static Data

**Goal**: Fully functional, styled, and accessible Teams listing page backed by placeholder data in `src/data/teams/teams.json`. No scraper work. No detail pages.

**Deliverables**:

1. `src/data/teams/teams.json` — 8–10 placeholder team entries covering all five age groups, both genders, multiple game nights and divisions. All fields populated including staff and training (for later use in P2).

2. `src/components/TeamTile.astro` — Astro component accepting a `team` prop. Renders:
   - Team name as the tile heading
   - Three pills: Division, Game night, Boys/Girls (each shows "TBC" when null)
   - The entire tile is an `<a href="/teams/{slug}">` — natively keyboard accessible, no `role="button"` hack needed
   - Visible focus indicator (`focus-visible:ring-2 ring-brand-gold ring-offset-2`)
   - Minimum 44px height enforced via Tailwind (`min-h-[44px]`)
   - `data-age-group`, `data-gender`, `data-game-night`, `data-division` attributes on the root element for the client-side filter to use

3. `src/pages/teams.astro` — Listing page:
   - Imports `BaseLayout.astro`
   - Loads `src/data/teams/teams.json` directly (Astro static import or `readFileSync` — static import preferred as file is committed)
   - Falls back to empty array if import fails, renders "No teams available" state
   - Groups teams by age group in order: u10s, u12s, u14s, u16s, u18s, Other
   - Sorts alphabetically within each group
   - Renders age group sections with `<h2>` headings
   - Includes the hero section (matches `scores.astro` / `team.astro` pattern)
   - Includes the filter bar (matches `team.astro` pattern: `sticky top-16 z-40 bg-white border-b`)
   - Filter options: All, Boys, Girls, then one button per age group (u10s–u18s), then one button per game night found in data, then one button per division found in data
   - "No teams match this filter" message with "Clear filter" button — hidden by default, shown via JS when all tiles are hidden
   - `<script>` block handles all filter logic (see below)
   - Filter state saved to `sessionStorage` on each filter change; restored on page load

**Filter JS logic** (in the `<script>` block of `teams.astro`):

```
State: { type: 'all' | 'gender' | 'ageGroup' | 'gameNight' | 'division', value: string }

On button click:
  1. Update aria-pressed on all buttons (true for active, false for others)
  2. Update visual active styles (bg-brand-purple text-white for active)
  3. For each .team-tile:
     - Read data attributes
     - Show/hide based on whether data matches the active filter
  4. For each .age-group-section:
     - Hide the section if all its tiles are hidden (avoids orphan headers)
  5. Show/hide the "no results" message
  6. Save { type, value } to sessionStorage key 'teams-filter'

On page load:
  1. Read sessionStorage key 'teams-filter'
  2. If present, simulate a click on the matching button
```

**Testing (P1)**:

Walk through each acceptance criterion from spec user stories 1–4:
- View the page: confirm age group order and alphabetical sort within each group
- Inspect each tile: confirm all three pills present; confirm "TBC" for any null values
- Click "Boys": confirm only boys teams visible; confirm age group headers with no matching teams are hidden
- Click "All": confirm full restoration
- Select "u14s": confirm only u14s teams
- Force-empty filter (e.g., a game night that has no teams): confirm "No teams match this filter" + "Clear filter" appears
- Tab through filter bar, press Enter: confirm filter activates; inspect `aria-pressed` attribute
- Click a tile: confirm navigation to `/teams/{slug}` (detail page will 404 until P2 — that is acceptable)
- Navigate back: confirm filter is still applied
- Resize to 320px, 640px, 1024px, 1440px: confirm layout matches NFR-008 grid spec
- Resize to mobile width: confirm filter bar scrolls horizontally (no clipping, no wrapping)

### Phase 2 — PlayHQ Integration and Team Detail Pages

**Goal**: Replace placeholder listing data with real PlayHQ data. Build the team detail pages showing fixture, ladder, staff, and training information.

**Deliverables**:

1. `scripts/scrape-teams.js` — New scraper script (detailed above). Writes `scripts/teams-data.json`.

2. Update `src/data/teams/teams.json` — Replace placeholder IDs with real PlayHQ team IDs after the first scrape run. Staff and training fields remain manually maintained here.

3. `src/pages/teams.astro` — Update data loading:
   - Try `readFileSync('scripts/teams-data.json')` (PlayHQ data, has fixture/ladder per team)
   - If file absent or parse fails, fall back to `src/data/teams/teams.json` (static placeholder)
   - Log to console: `console.warn('[teams.astro] teams-data.json not found; using placeholder data')`
   - Merge static file's `staff` and `training` fields onto each team by ID match

4. `src/pages/teams/[slug].astro` — Team detail page:
   - `getStaticPaths()` derives paths from `teams-data.json` (or falls back to `teams.json` slugs)
   - Each path receives its full team object as a prop
   - Merges staff and training from `teams.json` by ID
   - Renders inside `BaseLayout.astro`
   - Hero section: team name, age group, division, game night pills (same pills as the listing tile)
   - **Fixture section**: table with columns Date, Time, Opponent, Venue, Home/Away, Score
     - Upcoming games: score cells show "—"
     - Completed games: score cells show "Home – Away" format
     - Empty state: "No games scheduled" message
     - Data absent: "Team details are temporarily unavailable" notice
   - **Ladder section**: table with columns Pos, W, L, D, PF, PA, Pts
     - Phoenix row highlighted (border-l-4 border-l-brand-gold, matching `scores.astro` pattern)
     - Empty state: "Ladder not yet available"
   - **Staff section**: rendered only if at least one staff entry exists in the data file
     - Coach: "[Name]" — no email, no phone
     - Manager: "[Name]" — no email, no phone
     - Missing section (not blank label) when neither coach nor manager is present
   - **Training section**: "Training: [Venue] · [Day] @ [Time]"
     - Empty state: "No training scheduled" (not blank)
   - Layout: stacked vertically on mobile; fixture and ladder side-by-side on desktop (`lg:grid lg:grid-cols-2`)

**Error handling in `[slug].astro`**:
```
All null/undefined field accesses must use optional chaining or explicit null checks.
Missing field → display "TBC" or "—" (per spec), never a JS error.
Missing data file → show "Team details are temporarily unavailable" notice at the top of the page.
Log: console.warn('[teams/[slug].astro] Missing field:', { field, teamId, reason })
```

**Testing (P2)**:

- Run `node scripts/scrape-teams.js` — confirm `scripts/teams-data.json` is written
- Inspect JSON: confirm ageGroup, gender, gameNight, division are correctly parsed for at least 3 grade name variants
- Inspect JSON: confirm fixture and ladder are present for at least one team
- Navigate to a team detail page: confirm fixture table renders with date, opponent, venue
- Confirm completed games show scores; upcoming games show "—"
- Confirm ladder renders with Phoenix row highlighted
- Confirm staff section renders from `teams.json` when entries exist
- Confirm staff section is absent (not blank) when no entry exists for a team
- Confirm training section shows "No training scheduled" for a team with no training entry
- Delete `scripts/teams-data.json`; rebuild: confirm listing page falls back to placeholder and no unhandled errors; confirm detail pages render with placeholder notice

### Phase 3 — Game Detail Modal

**Goal**: Clicking a game row in the fixture table opens a modal with full game details.

**Deliverables**:

1. `src/components/GameModal.astro` — Modal markup, hidden by default (`hidden` class). Contains:
   - Overlay (`fixed inset-0 bg-black/50 z-50`)
   - Modal panel (full-width on mobile, max-w-lg centred on desktop)
   - Close button (X) in top-right corner
   - Content area: opponent, date, time, venue, home/away, score (if completed)
   - Focus trap implemented in the modal's `<script>` block

2. Update `src/pages/teams/[slug].astro`:
   - Add `data-game-id` attribute to each game row `<tr>`
   - Embed all game data as `data-*` attributes or as a JSON blob in a `<script type="application/json">` tag
   - Game rows become buttons (or the row has `role="button" tabindex="0"`) — consistent with NFR-005 (44px tap target)
   - On click/Enter: open modal, populate with game data from data attributes, trap focus
   - Escape key: close modal, return focus to the row that opened it
   - Outside click (overlay): close modal, return focus

**Modal JS behaviour**:
```
openModal(gameId):
  1. Read game data from data attributes
  2. Populate modal DOM fields
  3. Remove 'hidden' class from overlay
  4. Save reference to triggering element
  5. Move focus to modal panel (or close button)
  6. Begin focus trap (Tab/Shift+Tab cycles within modal focusable elements)

closeModal():
  1. Add 'hidden' class to overlay
  2. Return focus to triggering element

Escape listener: closeModal() on keydown 'Escape' when modal is open
Overlay click: closeModal() if click target is the overlay (not the panel)
```

**Testing (P3)**:

- Click a game row: confirm modal opens within 200ms with correct data populated
- Confirm modal shows score for completed game; omits score for upcoming
- Press Escape: confirm modal closes and focus returns to clicked row
- Click outside modal panel: confirm modal closes
- Tab through modal: confirm focus stays within modal (does not escape to page behind)
- On mobile (< 640px): confirm modal occupies full viewport height and is scrollable

## Complexity Notes

No constitutional violations. One item worth noting:

**Filter state in `sessionStorage` vs URL params**: The spec requires filter state to persist when the user navigates back. `sessionStorage` achieves this without changing the URL. If SEO-indexed filter URLs become a future requirement, the state mechanism can be switched to `?filter=boys` query params without changing any other part of the implementation — the data attributes and filter logic remain identical.
