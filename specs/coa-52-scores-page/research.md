# Research: COA-52 Scores Page

## Questions Evaluated

1. Should game details use a modal or a dedicated route?
2. Should weekly data be generated by extending existing scraper or creating a new one?
3. How should Sunday refresh be automated for a static Astro site?

---

## Decision 1: Details Surface

**Chosen**: Dedicated route (`/scores/[gameId]`) using shared generated data.

### Why
- Deep-linkable detail pages
- Better browser back/forward behavior
- Cleaner accessibility semantics than custom modal focus management

### Alternative considered
- Modal on `/scores`
  - Pros: faster interaction, fewer files
  - Cons: harder focus management; no stable URL

---

## Decision 2: Weekly Data Generation

**Chosen**: Generate a dedicated `weekly-games-data.json` artifact (via extension/wrapper around existing PlayHQ scraper logic).

### Why
- Clear separation from existing historical results/ladder model
- Stable contract tailored to COA-52 requirements
- Lower risk of regressions in unrelated score/ladder rendering logic

### Alternative considered
- Reuse and reshape `scores-data.json` only
  - Pros: fewer files
  - Cons: contract ambiguity; dual-purpose structure increases maintenance risk

---

## Decision 3: Sunday Refresh Automation

**Chosen**: Scheduled pipeline job (Sunday AEST) to run scraper + build/deploy.

### Why
- Meets auto-refresh requirement without app server
- Aligns with static-site deployment model

### Alternative considered
- Manual weekly refresh run
  - Pros: simplest setup
  - Cons: does not satisfy strict automation success criterion

---

## Operational Notes

- Keep API credentials in environment variables only.
- Log refresh success/failure with timestamp and reason.
- Provide manual trigger for emergency reruns.
